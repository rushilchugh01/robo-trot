# Robo Trot Learnings

## Phase Clock Mismatch Can Make a Good BC Policy Look Stuck

During BC/TXL checkpoint evaluation, the robot appeared to stand still or
twitch even though the teacher dataset contained walking-from-standstill
episodes and supervised action loss was decreasing.

### What Went Wrong

The teacher dataset stores `sin(phase), cos(phase)` in the observation. That
phase is generated by `FootspaceCPGIKTeacher.compute()` and advances with a
command-dependent gait frequency:

```python
freq = base_freq + (max_freq - base_freq) * clip(abs(vx) / speed_ref, 0, 1)
phase = phase + 2 * pi * freq * policy_dt
```

For `vx=0.3`, the teacher frequency is about `1.927 Hz`.

The evaluator was advancing phase with a fixed 1 Hz clock:

```python
phase = phase + 2 * pi * policy_dt
```

That made the policy see a gait rhythm that did not match the rhythm used in
training. The learned policy uses phase to coordinate swing/stance timing for
the trot, so the wrong phase rate produced mistimed leg actions and made the
robot look like it was trying to move but not entering a gait.

### Evidence

The dataset did include startup-from-standstill examples:

- At timestep 0, base velocity, joint velocity, and previous action are zero.
- Commanded forward velocity is nonzero.
- Teacher action labels are nonzero.
- By roughly 0.4 to 0.8 seconds, teacher rollouts reach walking velocity.

The same TXL checkpoint behaved very differently under different eval phase
clocks:

- Current broken eval clock, fixed 1 Hz at `vx=0.3`:
  - Mean forward velocity over 4 seconds: about `0.031 m/s`
  - Action RMS: about `0.149`
- Correct teacher-frequency clock at `vx=0.3`:
  - Mean forward velocity over 4 seconds: about `0.204 m/s`
  - Action RMS: about `0.216`
- Teacher controller at `vx=0.3`:
  - Mean forward velocity over 4 seconds: about `0.278 m/s`
  - Action RMS: about `0.248`

A corrected dashboard eval for TXL checkpoint `24200` produced:

- `eval_forward_velocity_mean`: about `0.234 m/s`
- Valid 8.33 second H.264 MP4 at
  `runs/bc_scratch_100k_v002/eval/media/step_000024200/txl_vx03.mp4`

### Fix

Checkpoint evaluation should advance phase with the same teacher CPG frequency
law used to generate the dataset:

```python
phase = advance_command_phase(phase, command, env.policy_dt)
```

The helper should:

- Keep phase fixed for true stand commands (`vx=0`, `yaw=0`).
- Use the teacher speed-dependent CPG frequency for walking commands.
- Preserve the observation contract between data generation and evaluation.

### Metrics To Trust

Do not trust aggregate reward alone for gait quality. Upright/alive/stability
terms can make a stationary robot look acceptable.

For locomotion quality, prioritize:

- `eval_forward_velocity_mean`
- Latest rollout MP4
- Action RMS / action magnitude
- Survival and fall rate
- Foot slip/contact quality
- Dataset action MSE only as an open-loop imitation diagnostic

### General Lesson

For BC/SFT locomotion, the eval loop must reproduce the teacher observation
contract exactly. Any generated observation feature, especially a gait clock,
must match the data generator. A small mismatch can turn a policy that learned
action labels into a closed-loop policy that appears stuck.
